What this is: a left-to-right node graph of every agent & script in the marketing pipeline (Steps 0→10). One column per step-group; arrows draw each producer→consumer edge derived from the data. Card color = wiring readiness (can data flow), not marketing quality. Green = ready · Amber = minor gaps · Red = blocked. Sourced from PART0 (flow spine), PART3 §5.2 (consumption matrix) and §6 (agent cards).

Source-of-truth precedence (the rule when sources disagree):

1. standards/ (marketing-soundness SPEC + register + directive) — governs decision quality.  ·  2. build-base architecture (PART0 / PART3 / PART1) + the seam field-map — the authority on scope, I/O and routing: what each step should produce and consume.  ·  3. the built bet-compiler SKILL — the template / bar for a "done" step.  ·  4. as-ran repo + old prompts — reference only: a pool of field-name vocabulary, proof of what's mechanically achievable, and a record of past failures.

Rule: when the as-ran (old prompts) and the build-base architecture disagree about what a step should produce or consume, the architecture wins. As-ran detail is seductive because it's concrete, but a lot of it is out-of-scope leftovers from when steps had different boundaries. Treat as-ran fields as candidates, adopted only when an architecture-defined consumer actually needs them. As-ran is evidence, not contract.

BLOCKER B1 (systemic) — no producer→consumer field map, no field-continuity checker

There is no central map of which field each step produces vs. which field the next step consumes, and nothing checks that they line up. Names and grains can silently drift across a seam.

Bites hardest at two hot seams: Step 2 → Step 3 (analyzed-record schema un-pinned, gap G1) and Step 3 → Step 5 (space-map output schema is an open slot; candidate-cell fan-out undefined, gap G3). Treat every cross-step arrow as unverified until a field-continuity checker exists.
