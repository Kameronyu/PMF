A structural fact governs this whole segment: **Reviewer A's own verdict was that every funnel/copy failure was DATA-SUFFICIENCY, not RULE-SOUNDNESS** ("The single recurring failure threading the whole segment is DATA-SUFFICIENCY, not RULE-SOUNDNESS: the prompts encode the LAW well; the run did not feed them proven material"). The redesign fixes the *prompt/topology* layer well. But data-sufficiency findings can only be CLOSED by guaranteeing the inputs exist *before* the consuming stage runs. So the live question across the board is: **does the new design make the missing input a hard refuse-precondition, or does it route the input to a downstream ○ build that is still open — leaving the same starved-input failure able to recur the first time the pipeline runs end-to-end before those builds land?**

### GAP A — S-4 "VOC as the foundation of copy" + Command+F register-verification UNRUNNABLE (FATAL at copy)
**Reviewer finding (Reviewer A / Funnel / S-4 + FA-4 + Copywriter):** quoted verbatim — *"COPY-DRAFT.md was produced with zero VOC and concedes it: 'NO VOC was run, so Spencer's Command+F verbatim check is unrunnable — word choices here are plausible, not verified against buyer language.' Spencer's Command+F is the explicit guilty-verdict test; the artifact admits it is unrunnable."* Verdict: **VIOLATION (FATAL at copy).**
**Operator-annotated?** YES — A37 ("We require one full session figuring out which inputs to capture in VOC, and bake it into the market selection framework and especially the funnel architect"); FA-4's inline annotation ("integrating VOC, figuring out inputs and outputs is genuinely gonna be such a huge deal during this build"). Strongly annotated → high propagation expectation.
**Version-status: LIVE (class can recur).** A whole VOC subsystem is added (§5), so the *structural* absence is being closed. But the FATAL charge was specifically that Command+F could not RUN because no corpus existed. The question is whether Command+F is now GUARANTEED runnable.
**Propagation: PARTIAL.** Mechanism IS named: §6.9 Copy Chief test #5 — *"Register check — script-assisted Command+F: flag copy vocabulary absent from the voc-bank/run corpus."* And §6.9 gates: *"voc-bank exists."* And Copywriter refuse: missing brief section fields. AND §6.5/§6.7 refuse conditions read *"missing voc-bank when run declares VOC active."* So the design DOES contain a refuse-condition that makes Command+F runnable **on paper**. BUT three load-bearing gaps keep it PARTIAL, not ADDRESSED:
  1. The refuse is conditioned on *"when the run declares VOC active"* (§6.5/§6.7) / *"soft-gate mode... explicit operator override"* (§6.5). This is the **exact D-08 soft-gate escape hatch that produced the as-ran VOC-absent run.** The design even names it: §6.5 "soft-gate mode stays available as an explicit operator override, never a silent default." The reviewer's FATAL finding survives one operator toggle — if the operator declares VOC inactive (as happened), Command+F is unrunnable again and the class recurs. The design has made the recurrence *visible and logged* (P9/P3) rather than *impossible*. That is a real improvement but it is not closure.
  2. The voc-bank's copy-bank content — what Command+F actually checks against — is **○ open on three counts simultaneously**: GAP-3 (per-PMBDE frequency table "named, not contracted"), GAP-5 (the phrase×frequency bank shape "isn't the documented contract"), and §5.3 ("Exact field schemas per product section... ○"). The Command+F test #5 cannot be specified until the bank it Command+Fs against is specified. So test #5 is a **named test against an unbuilt corpus shape** — structurally the same posture as the as-ran "unrunnable" check, just with the build now scheduled (Job 3) instead of skipped.
  3. Sequencing: §6.9 says "Build order unchanged from PART 2 Job 8," and §11 says VOC pass-1 is the MVP built first but pass-2/3 (the copy bank, the thing the copywriter RAGs) is "second." If the copywriter/chief build (Job 8) runs against a voc-bank whose pass-2/3 contract (GAP-3/5) is still open, test #5 degrades to a stub.
**Architecture location checked:** §5.1 Product 2, §5.2 (Copy Chief row: "Command+F that copy vocabulary exists in buyer language"), §5.3, §6.9 test #5, §6.5/§6.7 refuse conditions, §1.6 GAP-2/3/5, asran §9 (no contradiction — VOC genuinely was absent).
**Resulting flaw if not addressed:** The single FATAL copy charge recurs the first time the pipeline runs with VOC soft-gated off, OR runs Job 8 before Job 3's pass-2 bank contract lands. The Copy Chief emits a *green* register-check verdict that means nothing (it Command+Fs against an empty or pass-1-shaped corpus), reproducing the as-ran "plausible, not verified against buyer language" copy under a passing audit — strictly worse than the honest as-ran disclosure, because now a checker signs off on it. **Lands on: Copy Chief (test #5), Copywriter (RAG input), and the orchestrator's refuse-gate (the soft-gate toggle is the single point of failure).**

**Version-status: LIVE.** Congruency was never *enforced* in the as-ran run (the architect asserted "one angle" while the piece crossed drivers); a Funnel Auditor is now added (§6.8) explicitly to enforce it. The class is squarely live — this is a decision-level finding.
**Propagation: PARTIAL — and this is the audit's sharpest finding.** §6.8 names the enforcement point exactly: *"congruency checks per seam — one angle, one transformation, every section's belief install consistent with the chain."* That is a real, well-placed enforcement point (a cold auditor between architect and operator, mirroring the proven pipeline-audit pattern). **Architecture location checked:** §6.8 (auditor battery), §7.3 (registry `congruency` per-seam slot), §10 (congruency test content ○ Job 2), §7.2 (Funnel Architect row: "congruency-per-seam test (○ A33)").

### GAP D — FA-1 three-layer authority / never-merge: VALIDATED rule made CONFIGURABLE/vetoable (regression risk)
**Reviewer finding (Reviewer A / Funnel / FA-1 RULE-SOUNDNESS):** quoted verbatim — *"RULE-SOUNDNESS: SURVIVED — strongest single piece of reasoning in the segment. A faithful synthesis of the LAW... Refusing to merge non-comparable evidence matches Hormozi 'Actions-Only Reality Filter'... No violation."* This is the ONE thing Reviewer A most praised in the entire segment.
**Operator-annotated?** YES — P16 ("is the goal to NEVER merge, what does that even mean. I am just fearing whether or not slapping messaging on top of crowdfunding structure is proven... i want to challenge this a bit").
**Version-status: LIVE — but as a REGRESSION RISK, not a residual flaw.** This is the validated-keep-vs-redesign tension flagged in the brief. Both sides, per the evidence:
  - **The redesign demotes the rule.** §2 finding: *"P16 never-merge — NOT IN THE KB. Exhaustive search: no 'never merge' rule and no three-layer authority model exist anywhere in the KB. The architect skill's THREE-LAYER AUTHORITY MODEL... is prompt-local law with no KB provenance."* Consequently §6.7: *"The three-layer authority model / never-merge rule becomes a declared, configurable policy input — not hard-coded prompt law... the D5 session decides its fate and the architect consumes whatever policy that session writes (○)."* Veto register #8 confirms: "Never-merge becomes a configurable policy input to the architect (not deleted, not hard-coded) pending the D5 session."
  - **The reviewer validated the rule on first-principles LAW reasoning, independent of KB provenance.** Reviewer A derived it from Hormozi's Actions-Only Reality Filter and Carl's currency definitions — i.e. the rule can be sound as a reasoning principle *even though it has no dedicated KB file*. "No KB file names it" (PART3 §2) and "it is the strongest reasoning in the segment" (Reviewer A) are **not in contradiction** — they are about provenance vs soundness.
**The regression risk (surfaced, not resolved):** Making a reviewer-validated principle *configurable/vetoable* on the grounds that it lacks KB provenance risks the D5 session switching off (or weakening) the single thing the cold reviewer rated soundest — merging non-comparable currencies (e.g. treating crowdfunding raise as if it validated messaging, or an off-cell reputational mention as if it were Currency A). **Propagation: ADDRESSED-as-open (correctly).** The design does not silently drop the rule; it routes its fate to D5 with §2's provenance finding in hand (§10, §11 Job 4). That is the right handling. The audit's only addition: **the D5 session must be told this rule was the cold reviewer's highest-rated reasoning, so "no KB file" is not mistaken for "no support."** Provenance-absence is a reason to make it *explicit and examined*, not a reason to make it *optional by default*.
**Architecture location checked:** §2 item 2, §6.7 (configurable policy input), §10 (never-merge's fate ○ Job 4), §12 veto #8, PART1 P16.
**Resulting flaw if regressed:** If D5 weakens never-merge, the architect may merge crowdfunding/DTC/reputational evidence into one "proven" justification, directly reviving the FATAL FA-1 join finding (GAP B) the reviewer rated most damaging. **Lands on: D5 currency-model session + Funnel Architect (policy consumer).** This is a KEEP item (see §3), raised here so the configurability is not read as a free win.
