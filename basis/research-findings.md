## STREAM 1 — Making LLM output mechanically verifiable / self-checking

**Constrained decoding / enums (Outlines, dottxt-ai).** A JSON-schema `enum` or regex compiles to a finite-state machine; at each token the FSM hard-masks any token that would leave the valid set. Strongest possible membership test — category membership enforced *at generation time*, not checked after. Production-grade (integrated in vLLM etc.). Prompt-level shadow when you don't control decoding: give the model the closed list and require it to echo the chosen value verbatim from that list. URL: github.com/dottxt-ai/outlines. (Their "98% vs 76%" adherence numbers are marketing — directional.)

**Chain-of-Verification (CoVe).** 4-step self-check: (1) draft baseline answer; (2) plan verification questions; (3) answer each verification question *in isolation* from the draft; (4) revise, dropping items that fail. This IS a membership test: e.g. "list politicians born in NYC" → for each name generate "Was X born in NYC?" → answer in isolation → drop fails. Answering in isolation breaks the model's bias to defend its own draft. Liftable template: for every typed slot/list item force `CLAIM: <value> belongs to <category>. TEST: <defining question>. ANSWER (yes/no): __. EVIDENCE: __`. It's a prompting recipe — raises the floor, doesn't guarantee. URL: arxiv.org/abs/2309.11495.

**Instructor — exact-citations + llm_validator.** Most-adopted structured-output lib (~3M monthly downloads, de-facto standard). Two relevant mechanisms: (a) **exact-citations** — type a field as a substring that must be quoted verbatim from the source; a validator rejects the object if the quote isn't literally found in the context, triggering a re-ask. This is a cheap, MECHANICAL, non-LLM grounding check. (b) `llm_validator("must be supported by the context")` returns `{is_valid: bool, reason: str}`; on `is_valid=False` the `reason` is fed back as the re-ask message. Lift: every classification gets a paired `evidence_quote` that must be copy-pasteable from input; "if you can't quote the triggering text verbatim, you may not assign the label." URL: python.useinstructor.com/concepts/semantic_validation/.

**Guardrails AI — provenance validators.** `ProvenanceLLM` checks each generated sentence is supported by provided context (LLM-graded); `ProvenanceEmbeddings` checks embedding distance to source chunks; failures stripped or re-asked. Named concept: "provenance validator" / RAVG (Retrieval-Augmented Validated Generation) = generate, then validate each claim against its source before returning. The RAIL markup is legacy/clunky (mild over-engineering). URL: github.com/guardrails-ai/provenance_llm.

**Anthropic skills `skill-creator` — assertion/grading schema.** Official meta-skill (anthropics/skills repo ~115k stars). Mandates each eval has an `assertions` array; grading produces `grading.json` whose every entry uses exactly `{text, passed (bool), evidence}`. Rule to lift verbatim: **"For assertions that can be checked programmatically, write and run a script rather than eyeballing it."** That's a per-item quality predicate with mandatory evidence. URL: github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md.

**Self-Refine.** generate → self-FEEDBACK → REFINE → repeat, single model in all three roles. Key discipline: feedback must (i) LOCALIZE the specific problem and (ii) give a CONCRETE instruction to fix — not "looks good." ~20% avg gain (task-dependent, not a guarantee). Antidote to presence-checking. URL: arxiv.org/abs/2303.17651.

**Self-verification rubric ("score each dimension, revise if < threshold").** Practitioner pattern (Anthropic docs endorse "run output against a short checklist matching your success criteria"): append a block where the model scores its draft 0–5 per dimension (Correctness / Completeness / Clarity / Actionability) and MUST revise+rescore if any dimension < threshold. Converts vague "is it good?" into per-dimension testable predicates with a hard gate. Folk wisdom but standard.

**Constitutional-AI critique-revision.** Sample response → model critiques it AGAINST AN EXPLICIT NAMED PRINCIPLE → model revises per the critique. The critique is always tied to a named principle, never generic. Lift: each completeness check references a named rule (A1/A2/...) so critique is anchored, not vibes. URL: arxiv.org/abs/2212.08073.

**BAML (BoundaryML) — Schema-Aligned Parsing.** Schema-first DSL that parses messy LLM output (markdown-in-JSON, trailing commas, CoT-before-answer) into typed objects. Relevant: SAP lets you keep a reasoning/justification/`source` field ALONGSIDE each typed field in one schema without breaking the parser — so demand a `source` per field and still get a clean typed object. Co-locate `value`+`source` per field; don't make provenance a trailing afterthought. (Vendor "2–4× faster" numbers unverified.) URL: github.com/BoundaryML/baml.

**pydantic-ai — output_validator + ModelRetry.** Raise `ModelRetry("specific message")` from a validator to bounce output back with the error; configurable retry budget. Lift: a validator that, on a missing/ungrounded source tag, returns a SPECIFIC retry message rather than silently passing. URL: ai.pydantic.dev/output/.

**TOP-5 most liftable (stream 1):** (1) CoVe per-item membership template, answered in isolation. (2) skill-creator's `{text,passed,evidence}` triple + "if scriptable, write the script, don't eyeball it." (3) Instructor verbatim-substring citation field (mechanical, no 2nd LLM call). (4) 0–5-per-dimension rubric with a hard revise gate. (5) Co-locate value+source per field; validator raises a specific retry on missing/ungrounded source.
**Skeptic note:** only constrained decoding and verbatim-substring checks are TRULY mechanical; CoVe/rubric are prompt scaffolds that raise the floor. Use mechanical checks for hard contracts, prompt patterns as the self-check scaffold around them.

## STREAM 2 — Retry / escalation / "try-harder" loops (the Closer pattern, B3/B4)

**Evaluator–Optimizer (Anthropic "Building Effective Agents").** Canonical industry taxonomy (cookbook ~45k stars). "One LLM generates a response while another provides evaluation and feedback IN A LOOP." Optimizer produces candidate → evaluator critiques against EXPLICIT CRITERIA → if rejected, feedback fed back, optimizer regenerates → loop until evaluator approves. Fit when "a human can articulate the feedback and the LLM can provide such feedback." Terminates on evaluator PASS or a max-iteration stop. Their search example: "the evaluator decides whether further searches are warranted" (a B4 analog). Lift: two-role split (our Closer = optimizer driven by a gate); explicit-criteria gate as loop condition; use the named pattern as vocabulary. URL: anthropic.com/engineering/building-effective-agents.

**Orchestrator–Workers (Anthropic).** "A central LLM dynamically breaks down tasks, delegates to worker LLMs, and synthesizes their results." Subtasks determined at runtime, not pre-defined. Use case: "gathering and analyzing information from multiple sources." Maps to B4: the 3–5 mini-search fan-out is an orchestrator-workers instance — each worker gathers one slice, orchestrator synthesizes. Async reference notebook exists. 

**Prompt chaining with GATES + Voting (Anthropic).** Decompose into fixed steps; "add programmatic checks (a 'gate') on any intermediate step to ensure the process is still on track." The gate is exactly our trigger: detect a blank/failed field → branch to the Closer instead of passing a bad artifact downstream. **Voting** variation: run the same task multiple times for diverse outputs, aggregate programmatically (cheap B4 confidence boost).

**Reflexion.** Strongest academic match for the Closer (~3.2k stars; re-implemented in LangGraph/smolagents). Three roles: **Actor** (produces attempt), **Evaluator** (scores via task feedback / unit tests / env reward), **Self-Reflection** model (converts the failure signal into a TEXTUAL lesson). The reflection is stored in an episodic memory buffer and PREPENDED as context on the next trial, so the agent retries WITH explicit knowledge of why it failed. Configurable `ReflexionStrategy`: NONE / LAST_ATTEMPT / REFLEXION / LAST_ATTEMPT_AND_REFLEXION = how much prior-attempt context the retry agent receives. Bounded by `num_trials`. Lift: our "log what you tried" becomes STRUCTURED retry-context fed to the next attempt, not just an audit trail; episodic-memory-of-failures; num_trials as the bound. URL: github.com/noahshinn/reflexion.

**Self-Refine (again, for B3).** Proves ONE prompt / same model can both produce and repair — justifies a same-system-prompt Closer. Requires feedback be specific+actionable so the next pass is actually better. Stops on a "good enough" signal OR iteration cap.

**CRITIC.** Tool-interactive self-correction. KEY FINDING: **"LLMs alone cannot reliably critique/correct their own work — external tools are essential."** Output → interact with external tools (search, code interpreter, calculators) to VERIFY specific claims → revise based on tool-grounded feedback → repeat until verified or max iters. Lift: the Closer must fill the blank "by any means" using TOOLS/external lookups, NOT by re-reasoning from priors. Justifies why B3's "by any means" = tools and why B4 gathers RELATIVE EVIDENCE rather than asserting. URL: arxiv.org/abs/2305.11738.

**Self-Consistency.** Sample N diverse chains-of-thought → majority/plurality vote on the answer. Higher agreement = higher confidence. Bound = fixed sample count N. Maps to B4: "3–5 mini-search agents" is self-consistency applied to evidence gathering — run several independent passes, trust the convergent answer over a single assertion. URL: arxiv.org/abs/2203.11171.

**DSPy Assert/Suggest → Refine (the most precise B3 model).** On a failed validation check, instead of crashing it triggers BACKTRACKING + RETRY with dynamic signature modification — it injects two new fields into the next prompt: **`Past Output`** (the prior output that failed) and **`Instruction`** (a user-defined message saying what went wrong + what to fix). Termination distinction we need: **`Assert` (HARD)** retries up to `max_backtracks` (default 2); if still failing, HALTS with an error → the genuinely-critical field that must not pass downstream broken. **`Suggest` (SOFT)** retries up to the same cap, but on exhaustion LOGS the persistent failure and CONTINUES (best-effort) → exactly B3's "exhausted every approach (logging what it tried)" then proceed. Lift: (1) Past Output + Instruction injection on retry; (2) max_backtracks as the bound; (3) hard(halt) vs soft(log-and-continue) dichotomy. HYPE FLAG: `Assert`/`Suggest` are DEPRECATED — cite the mechanism, use `dspy.Refine`/`dspy.BestOfN` if implementing. URL: dspy.ai/learn/programming/7-assertions/.

**LangGraph — conditional edges, loops, Reflexion impl.** Agent logic = a state graph; nodes are functions; CONDITIONAL EDGES route based on state. A loop = a conditional edge pointing back to an earlier node: "if execution fails OR a retry limit is reached, end; otherwise return to regeneration." Their Reflexion impl: Responder (initial answer) → Execute Tools (invoke tools to CHECK claims) → Revisor (original answer + tool outputs + critique → revised answer), cycled. Lift: conditional-edge-with-retry-counter as the control-flow primitive; Responder→ExecuteTools→Revisor as a tool-grounded retry loop. URL: langchain.com/blog/reflection-agents.

**CrewAI — task guardrails + guardrail_max_retries.** A guardrail (Python fn OR LLM-based NL check) validates a task's output before it passes downstream. On failure, the system catches it, generates an error message, and sends it back: "You failed this specific rule. Try again." Guardrail returns `(True, validated_result)` or `(False, "error explaining the failure")` — that error string becomes retry context. Bounded by `guardrail_max_retries` (e.g. =3). Lift: the `(bool, message)` return contract; the explicit "you failed THIS, try again" feedback. URL: docs.crewai.com/en/concepts/tasks.

**OpenAI Agents SDK — guardrails (tripwires) + handoffs.** Input/output/tool guardrails validate; on failure raise a TRIPWIRE exception that HALTS the agent. Tool guardrails can `reject_content("...try again message...")` to send a correction back WITHOUT fully halting (soft retry). HANDOFFS transfer the FULL message history to another agent, which "sees the entire conversation as if it had been participating from the start." Lift: handoff-with-full-history = our "hand the failing artifact to ANOTHER agent (same context)"; tripwire = the gate for truly-critical fields. URL: openai.github.io/openai-agents-python/guardrails/.

**AutoGen / AG2 — nested chat, society-of-mind.** Nested chat spins up a SEPARATE inner conversation for a subtask, with an LLM REFLECTOR analyzing the whole inner history and returning a single result to the outer chat. SocietyOfMindAgent runs an internal group chat but "appears to the external world as a single agent." Lift: encapsulate the Closer's whole try-harder loop behind a single interface — outer pipeline just sees "field filled / field exhausted"; max_round as bound. 

**GPT-5 / 5.1 persistence prompting (the verbatim phrasing for the Closer).** Canonical: *"You are an agent — please keep going until the user's query is completely resolved, before ending your turn and yielding back to the user."* / *"Persist until the task is fully handled end-to-end within the current turn... do not stop at analysis or partial fixes."* HOW THEY BOUND IT (critical): pair persistence with EXPLICIT STOP CONDITIONS — "clearly state the stop conditions... define when, if ever, it's acceptable to hand back," and a hard TOOL-CALL BUDGET ("limiting the number of tool calls, for example to two"). Lift: adapt to "Do not end your turn until `<field>` is filled or you have demonstrably exhausted every available approach," paired with a max-attempt/tool-call budget and an explicit definition of "exhausted." URL: cookbook.openai.com/examples/gpt-5/gpt-5_prompting_guide.

**Bounding non-termination (universal finding).** NOBODY ships an unbounded "try forever" loop. The universal bound is one of: `max_backtracks` / `guardrail_max_retries` / `num_trials` / `max_round` / `max_iterations`, OR a tool-call budget. Anthropic: include "stopping conditions (such as a maximum number of iterations) to maintain control." Operationalize "exhausted every approach" as a CONCRETE budget (N attempts or N tool calls) + a structured log of what each attempt tried — never the model's judgment alone.

**TOP-5 most liftable (stream 2):** (1) DSPy Assert-vs-Suggest hard/soft + Past Output+Instruction injection. (2) Reflexion episodic-memory-of-failures fed to next attempt, bounded by num_trials. (3) Anthropic Evaluator-Optimizer + Orchestrator-Workers + the "gate" concept. (4) GPT-5 persistence phrasing bounded by stop-conditions + tool-call budget. (5) CRITIC: self-correction must be TOOL-GROUNDED not priors-based.

## STREAM 3 — IO contracts, handoffs, lanes, provenance, deterministic orchestration

**OpenAI Agents SDK — handoff() input_type + input_filter.** Handoff exposed to the model as a tool (`transfer_to_<agent>`). `input_type` (a schema) becomes the tool's parameters; SDK validates the returned JSON locally and passes the parsed object to an `on_handoff` callback. CRUCIAL RULE: `input_type` is for SMALL model-generated METADATA only — `reason`, `priority`, `summary` — NOT application state (that travels out-of-band in a context object). `input_filter` receives the history and returns a filtered version — e.g. built-in `remove_all_tools` strips tool-call noise before the next agent sees it. A `RECOMMENDED_PROMPT_PREFIX` constant teaches agents how handoffs work. Lift: (1) metadata-vs-state split as a hard rule — "handoff payload = model-decided metadata only; durable state out-of-band"; (2) input_filter as a normalization gate between steps; (3) ship a canonical prompt fragment that teaches the lane boundary. URL: openai.github.io/openai-agents-python/handoffs/.

**LangGraph — typed State, channels, reducers, Command handoffs.** State is a typed schema; each field is a "channel." Updates merge by REDUCERS declared via `Annotated[type, reducer]`, signature `(Value, Value) -> Value`. Append-fields use `operator.add` or `add_messages` (dedupes by ID); scalar fields default to OVERWRITE. The schema is the input contract to every node; nodes emit partial updates applied through the reducer. `Command(goto=target, update={...})` both ROUTES and WRITES typed state in one object. Lift: the reducer-per-field model is the cleanest formalization of "a multi-mode field needs a selection rule" (C1) — every field declares HOW it combines (overwrite/append/custom). Borrow vocabulary: "channel," "reducer." URL: reference.langchain.com/python/langgraph/graph/state/StateGraph.

**Pydantic-AI — deps_type + output_type generics with validation-retry.** `Agent[DepsType, OutputType]` is generic in BOTH injected dependencies and result. `deps_type` injects DBs/clients/config type-safely; `output_type` (a model) becomes JSON schema the model must follow; VALIDATION ERRORS ARE FED BACK to the LLM with a retry request (configurable max retries) — self-correcting contracts. Lift: validation-error-feedback-retry as the ENFORCEMENT TEETH behind a contract (not "ask nicely" — reject + retry on schema violation); deps-vs-output generic split = input/output hygiene. URL: ai.pydantic.dev/agent/.

**CrewAI — output_pydantic task contracts.** `Task(output_pydantic=MyModel)` forces output into a schema, validates, retries on bad JSON; result available as a real validated object. CAVEAT/hype: known issue where the schema wasn't always injected into the system prompt — enforcement has had gaps. Cite as corroboration, not foundation.

**OpenAI Swarm (educational only).** Two primitives: **routines** (NL instructions + tools) and **handoffs** (a function that returns the next agent). Cleanest one-liner: "an agent is a system prompt plus a list of functions." HYPE FLAG: explicitly an educational toy, superseded by the Agents SDK — cite the concept, not as production prior art.

**Anthropic "Building Effective Agents" — routing, sectioning, separation of concerns.** Five composable patterns: prompt chaining (with "gate" checks), routing ("separation of concerns... optimizing for one input shouldn't hurt another"), parallelization incl. **SECTIONING** — one model answers while a SEPARATE model screens for policy, explicitly "performs better than having the same LLM call handle both" — orchestrator-workers, evaluator-optimizer. Core principles: simplicity, transparency (show planning steps), a well-engineered agent-computer interface. Lift: sectioning is the STRONGEST authority for one-job-per-agent lanes (guardrail-in-separate-call beats combined call); the "gate between chained steps" is the mechanical-check pattern. URL: anthropic.com/engineering/building-effective-agents.

**Google ADK multi-agent patterns — scoped roles/tools/permissions.** "An agent's scope is limited to its assigned role, narrower focus, smaller set of tools, a specific goal." Executors get SCOPED tool access and SCOPED WRITE permissions (write limited to a branch/dir; read-only to prod by default). "Codify handoffs with owners and timeouts; keep context bounded so agents don't step on each other." Lift: lane scoping enforced by TOOL/PERMISSION restriction, not only prose (maps to skills' `allowed-tools`); "every handoff has an owner" supports "every emitted field has a named consumer."

**Anthropic Citations API — guaranteed-valid cited_text.** Set `citations.enabled=true` on document blocks; documents chunked into sentences; Claude returns interleaved citation blocks where `cited_text` is GUARANTEED a valid verbatim pointer into the provided documents (EXTRACTED by the system, not generated by the model) + a content-block index. `cited_text` doesn't count toward output tokens. IMPORTANT CONSTRAINT: citations INTERLEAVE blocks and therefore CONFLICT with strict JSON-schema structured output (enabling both returns a 400). Lift: system-extracted verbatim spans = gold standard for provenance; but for TYPED output, use a prompt-enforced verbatim-quote field inside your own schema, not the native API. URL: platform.claude.com/docs/en/build-with-claude/citations.

**"FullCite" / citation-enforced RAG.** Rule: "every claim in the answer is followed by a citation comprising the document identifier AND a verbatim evidence span." Two strategies: simultaneous (cite during generation) vs post-hoc retrieval. Promptfoo has a red-team plugin that TESTS whether outputs are actually attributable (enforcement-by-eval). Lift: "each fact → doc-id + verbatim span" as literal provenance spec wording; an automated attribution check in the pipeline. URL: [unverified arxiv id — likely mis-transcribed; do NOT reuse].

**Cross-vendor strict structured outputs (strict:true).** `response_format: {type:"json_schema", json_schema:{...}, strict:true}` constrains decoding so output is guaranteed schema-valid (supply Pydantic/Zod). Lift: make provenance + caveat REQUIRED schema fields so omission is a hard validation failure — strict mode is the enforcement teeth (subject to the citations conflict above).

**Input hygiene — HTML/boilerplate stripping before the classifier.** Two-tier: trafilatura first to drop nav/ads/boilerplate while preserving main content; fall back to html-to-markdown for structure-sensitive docs; BeautifulSoup/regex for tags. Finding: GPT-4o-class models lose LITTLE accuracy when HTML structure is stripped — feeding the classifier a minimal clean slice is safe and cheaper. Caveat: over-cleaning strips signal; tune. This is DETERMINISTIC CODE, not an LLM call — do it in a script between steps. URL: github.com/shubhamofbce/clean-html-for-llm.

**Deterministic DAG orchestration.** Tasks as a DAG with strict dependencies; each task IDEMPOTENT (same input → same output, no side effects, safe to re-run). Honest finding: LangGraph "is a DAG even with no LLM" but LACKS scheduling, built-in retries/backoff, backfills, distributed execution — those live in Airflow/Prefect/Dagster. Dagster recommended for asset-tracked multi-step RAG. "Prompt2DAG" shows spec → executable DAG via DETERMINISTIC TEMPLATE EXPANSION (preferred over LLM synthesis for reliability). Lift: (1) idempotency as a hard requirement per `claude -p` step; (2) use a real orchestrator (Make/Prefect/Dagster) for scheduling/retry/backfill, reserve LLM-graph frameworks for in-step logic; (3) deterministic template expansion beats LLM-generated DAG → favor a fixed DAG/Makefile over a self-planning orchestrator. URL: arxiv.org/html/2509.13487v1.

**"Agent-as-a-function."** Anthropic's workflow-vs-agent split: a WORKFLOW is deterministic predefined code paths (an agent invoked as a pure function with typed in/out); reserve "agent" for dynamic self-direction. OpenAI's `Agent.as_tool(parameters=...)` exposes a sub-agent as a TYPED CALLABLE that returns structured output WITHOUT transferring the conversation — cleanest "agent as a function" primitive: call a scoped sub-agent like a function, get a typed result, retain control.

**Skill-builder prior art.** **Anthropic Agent Skills** (open standard as of Dec 2025): a skill = directory with `SKILL.md` (YAML frontmatter `name`+`description` preloaded into system prompt = level-1 disclosure; body = level-2; bundled `reference.md`/scripts = level-3, loaded ONLY on demand). Best practices: "start with evaluation"; split files when SKILL.md gets unwieldy; keep body lean (recurring token cost); "code in a skill provides DETERMINISTIC, repeatable behavior" where token-generation is unreliable. **Claude Code skills/subagents:** `allowed-tools` frontmatter restricts what the skill can do while active (enforced lane); `context: fork` runs the skill in an ISOLATED subagent context window. Lift: progressive-disclosure file layout; description carries all when-to-use triggers; bundle deterministic steps as scripts; `allowed-tools` = structural lane enforcement; `context: fork` = minimal isolated context per step. URLs: agentskills.io; github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md.

**TOP-5 most liftable (stream 3):** (1) Anthropic sectioning (separate screener vs answerer) for lanes; chained "gates" for mechanical checks. (2) Citations API system-extracted verbatim spans for provenance (+ the strict-JSON conflict). (3) LangGraph typed State + reducers (overwrite/append/custom) for multi-mode fields. (4) OpenAI handoff input_type/input_filter — metadata-vs-state rule + remove_all_tools. (5) FullCite "every claim → id + verbatim span" + Promptfoo attribution check.
**HYPE FLAGS:** Swarm = educational only. CrewAI output_pydantic real but has schema-injection gaps. LangGraph-as-full-orchestrator over-claimed — lacks scheduling/retry/backfill; use for in-step logic, not as your scheduler.

## STREAM 4 — Positive specification vs negative constraints

**Vendor consensus (authoritative, practitioner-tested).** Anthropic: explicit named principle "Tell Claude what to do instead of what not to do" — replace "Do not use markdown" with "Your response should be smoothly flowing prose paragraphs." Claude 4.x follows instructions more literally, so positive direction matters more. OpenAI: "Instead of saying what not to do, say what to do instead" — reframing forces the specificity that produces good output. Google: positive framing beats negation (esp. image gen): "no blur" → "crisp sharp focus." All three converge.

**OpenAI GPT-4.1 guide — three load-bearing points.** (1) Rigid rules can BACKFIRE: "instructing a model to ALWAYS follow a specific behavior can occasionally induce adverse effects" — e.g. "you must call a tool before responding" causes hallucinated/null tool inputs; the fix is a POSITIVE ESCAPE CLAUSE ("if you don't have enough info, ask the user"), not another prohibition. (2) Conflict resolution by POSITION: the model "tends to follow the instruction closer to the END of the prompt" — long constraint lists create internal conflicts silently resolved. (3) Examples must be backed by rules and vice-versa: "ensure any important behavior demonstrated in your examples is also cited in your rules."

**Google Gemini guide — negatives over-index + placement.** "Open-ended system instructions like 'do not infer' or 'do not guess' may cause the model to OVER-INDEX on that instruction and fail to perform basic logic/arithmetic" — fix is positive (tell it to use provided context for deductions). Placement: models "may DROP negative constraints if they appear too early" — place a critical restriction as the FINAL line.

**"White Bear: Ironic Negation in Transformer Models" (2025).** Empirical/mechanistic. Two controlled experiments over a 5,000-prompt benchmark (ReboundBench) + attention-head circuit tracing. Finding: "Negation instructions such as 'do not mention X' can PARADOXICALLY INCREASE the accessibility of X" — to follow "do not mention X" the model must internally activate X. (a) rebound "consistently arises immediately after negation" and "intensifies with longer or semantic distractors" (worse under context load); (b) repetition-type filler supports suppression while SEMANTIC load drives rebound; (c) sparse middle-layer attention heads amplify the forbidden token while early layers suppress. Nuance: not pure backfire everywhere — copy-suppression heads CAN suppress, so the effect is conditional — which matches "case-by-case, only for strong attractors," not a blanket ban. Preprint. URL: arxiv.org/abs/2511.12381.

**16x Eval — "The Pink Elephant Problem."** Practitioner synthesis (well-sourced). Real failures of negative rules: Claude Code creating `file-fixed.py` despite a "NEVER create duplicate files" rule; users reporting worse output "the more DO NOTs are included." Endorses the exact balance: "positive framing provides clearer guidance, but negative constraints still have their place — for preventing unethical/harmful behavior, firm boundaries." Tips: "use them sparingly, be specific, pair with positives, make them absolute." Observes the best-written system prompt (Anthropic's) phrases boundaries as DESCRIPTIVE statements ("Claude does not...") not imperative "DO NOT." URL: eval.16x.engineer/blog/the-pink-elephant-negative-instructions-llms-effectiveness-analysis.

**Spec-driven / acceptance-criteria generation.** Thesis maps onto "precise target + membership test": "Structured specs constrain the search space that makes unconstrained generation chaotic; acceptance criteria give the agent AND its verifier something checkable." Each task ends with a verification step ("did the output satisfy the acceptance criteria? if not, iterate — bounded by the spec, not free-running").

**Structured-output / controlled vocab (empirical).** A required schema "constrains the model to that logical boundary," eliminating whole categories of fabrication; classification with a controlled vocabulary / enum restricts the model to a fixed allowed set. One reported deployment cut error 14%→2% with constraint-based generation; enums suppress "creative" values like "Billing Issue (urgent)." This is the mechanism behind "prove the value fits the category" rather than trusting a label (rule A1).

**Few-shot / show-don't-tell.** For format/classification, 2–3 demonstrations "significantly outperform" zero-shot rule statements; the label space + input distribution shown in demos drive performance ("even random labels beat no labels"). Caveat: examples cost tokens/context length — use a FEW precise examples, not many.

**Instruction bloat / long-context degradation.** "Lost in the Middle" (TACL, widely replicated): U-shaped positional performance — accuracy highest when relevant info is at start or end, degrades >30% in the middle. "Context Rot" (Chroma, 18 models): every model degraded with input length, NON-UNIFORMLY, well before the window limit (a 200K window can lose accuracy at ~50K); even with distractors masked, accuracy fell ~7.9% from length alone; semantic similarity of distractors drives decay more than raw length. Multi-constraint benchmarks (ComplexBench/CFBench/FollowBench): compliance drops as simultaneous constraints rise (notably past ~10); models "struggle with constraints involving numbers, NEGATIONS, or long instructions." URLs: arxiv.org/abs/2307.03172; research.trychroma.com/context-rot.

**VERDICT.** Weight of evidence clearly SUPPORTS "positive-first; negatives only for strong documented attractors; keep them to one line, at the end." Convergent vendor guidance + mechanistic evidence negatives are risky + direct support that a positive checkable boundary (schema/enum/membership test) beats a prohibition (the 14%→2% case) + "minimal" support (every line costs accuracy, buried constraints get dropped). NUANCE (not refutation): negatives still belong for hard safety/firm boundaries; when kept, make them absolute, specific, ONE line, at the END; phrase boundaries as positive/descriptive ("X does not...") not imperative "DO NOT." THIN spots: almost no clean head-to-head RCT isolating positive vs equivalent negative on the same task with frontier chat models; strongest paper (White Bear) is a preprint; "membership-test as positive boundary" framing is our synthesis (reasonable, novel); "one line" is an inference from Context Rot + constraint-count degradation, not directly measured.
