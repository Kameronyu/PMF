# Funnel Deep Pass — Router + Section Analyzer (agent prompts + schema + controlled vocab)

Goal: From one cleaned funnel package, classify the routing scope (Router), then emit
belief-instance records that capture what each section of the funnel DOES — the beliefs
it installs, how it installs them, and the verbatim copy it uses. Output feeds the future
birdseye synthesis agent.

**Pipeline (2 judgment agents, with deterministic scaffolding around them):**

```
[funnel_package per competitor]             ← from funnel-assemble.js (Plan 03-01)
   ↓
[funnel-clean.js]                           ← section-marked verbatim copy (deterministic)
   ↓
[funnel-score.js]                           ← validation_lane + validation_strength (deterministic)
   ↓
Router (AGENT 1, cheap model)               ← sets routing_flag BEFORE the analyzer
   → funnel-level routing_flag stamped
   ↓
funnel-analyzer-context.js (assembler)      ← assembles [DR bundle (via inject-dr.js)] + [cleaned funnel body in <funnel_copy>] into the analyzer's spawn prompt
   ↓
Section Analyzer (AGENT 2, quality model)   ← the judgment workhorse; emits belief records
   → N belief-instance records per funnel
   ↓
validate-analyzer.js (PostToolUse hook)     ← rejects off-contract output (deterministic)
   ↓
[funnel-store.js]                           ← structured JSON under runs/<space>/funnels/ (deterministic)
```

**Determinism principle:** Anything that can be deterministic is pulled OUT of the agents.
The Router makes one classification (transformation-similar or not). The Analyzer makes
judgment calls on belief segmentation and tagging. The hook rejects bad output — it does NOT
trust the prompt.

## SCHEMA (the contract every stage reads/writes)

### 6a. Funnel-level fields (recorded ONCE per funnel)

```json
{
  "funnel_id": "string — unique, generated by assembler",
  "competitor": "string — from PMF stage (brands.json slug)",
  "source_type": "string — 'dtc' | 'crowdfunding' — from PMF stage, do not re-derive",
  "transformation": "string — from PMF stage chosen cell; the ONE transformation this funnel sells; do not re-derive",
  "niche": "string — from PMF stage; confirms proven spend is for our niche",
  "routing_flag": "string — 'structure_only' | 'messaging_full' | 'both' — set by Router at plan time, NEVER inferred by Analyzer",
  "primary_claim": "string — verbatim headline claim the funnel leads with; set by Analyzer",
  "claim_type": "string — direct | enlarged | mechanism | enhanced — the funnel top-line claim type; set by Analyzer; REUSES CLAIM_TYPE_ENUM",
  "awareness_entry": "string — where on the awareness ladder the funnel opens (unaware → most-aware); read off problem-education load: count sentences before product is named; is the problem explained or assumed; set by Analyzer",
  "funnel_sequence": ["string — ordered list of pages/sections and what each does; the container skeleton; set by Analyzer"],
  "offer_mechanic": "string — the commitment hook; e.g. 'deposit / tiered pledge / early-bird / founder pricing / bundle / guarantee'; set by Analyzer",
  "urgency_construction": "string — the mechanic AND the number (e.g. 'first 200 at $X then steps to $Y, countdown timer visible above pledge button'); NOT just 'they used urgency'; set by Analyzer",
  "validation_lane": ["string — 'A' | 'B' | 'unknown' — array; set by funnel-score.js"],
  "validation_strength": {
    "_note": "set by funnel-score.js — currency-appropriate numbers per lane",
    "currency_a": {
      "max_run_duration_days": "int | null",
      "impression_bucket_multiplier": "int 1-8 | null",
      "variant_count": "int | null",
      "composite_rank_score": "number | null",
      "meets_anti_fluke_floor": "boolean | null"
    },
    "currency_b": {
      "amount_raised": "string | null",
      "backer_count": "int | null",
      "funded_vs_failed": "'funded' | 'failed' | null",
      "delivered_vs_not": "'delivered' | 'not_delivered' | null",
      "_delivered_signal": "string | null"
    }
  }
}
```

### 6b. Belief-instance record (N per funnel — one per belief installed or reinforced)

```json
{
  "funnel_id": "string — links to parent funnel",
  "position": "int — funnel-level ordinal (1st, 2nd, 3rd belief-instance encountered across the whole funnel); NOT page-local; see §5 / D-12",
  "belief_id": "string — one of the 9 anchors (preferred) or a new proposed belief_id if none fits",
  "belief_confidence": "'high' | 'low' — LOW for any belief_id that is NOT one of the 9 anchors (overflow beliefs must be flagged for review)",
  "belief_kind": "'crowdfunding-specific' | 'general-dr' — see BELIEF_KIND_ENUM; which category of belief this is; crowdfunding-specific = beliefs that only appear in crowdfunding funnels (e.g. it-will-ship, trust-the-brand-or-founder as founder-credibility); general-dr = beliefs that appear in any DR funnel",
  "execution_type": "string — see EXECUTION_TYPE_ENUM below; HOW the belief is installed",
  "execution_detail": "string — GRANULAR free text; specific enough that the sub-claim is recoverable; NOT 'they build trust' but 'founder names the exact factory, shows a dated production photo, and states he placed the full MOQ order himself — installing ship-confidence via founder financial skin-in-game'",
  "proof_tier": "string — 'Tier1' | 'Tier2' | 'Tier3' — see PROOF_TIER_ENUM",
  "moves": ["string — see MOVE_ENUM; discrete persuasive moves in this section"],
  "verbatim_refs": [
    {
      "text": "string — exact verbatim substring from the cleaned funnel copy; must be a verbatim substring (hook-enforced)",
      "source": "'competitor-marketing' | 'review_language' — whether this verbatim is from the brand's own marketing copy or from an on-page review/testimonial"
    }
  ]
}
```

### Closed enums (a value off-list is a hard reject)

```
CLAIM_TYPE_ENUM:        direct | enlarged | mechanism | enhanced
                        (REUSED from step1-light-pass.md — do NOT coin a parallel vocab)
                        direct    = bare outcome promise ("blocks distractions")
                        enlarged  = specified/quantified/conditional ("blocks distractions for 4+ hrs")
                        mechanism = outcome tied to named how/why ("via an amber e-ink display that removes app access")
                        enhanced  = stacked superlative/differentiator ("the ONLY e-ink device clinically proven to…")

EXECUTION_TYPE_ENUM:    mechanism-explanation | feature-as-evidence | demo | authority |
                        social-proof | founder-credibility | risk-reversal | scarcity |
                        urgency | exclusivity | story-epiphany | comparison |
                        consequence-of-inaction
                        (from Carl Weische's six persuasion elements, VSL 8-stage narrative,
                        funnel-architecture pre-sale 4-part framework, and differentiator believability tiers;
                        resolved against repo conventions — no duplicate tags)

PROOF_TIER_ENUM:        Tier1 | Tier2 | Tier3
                        Tier1 = self-evident / demo (visible proof needing no external validation)
                        Tier2 = authority (expert, credentials, clinical study, press logo)
                        Tier3 = social-proof (testimonials, reviews, UGC, named buyer results)
                        (from differentiator framework believability tiers)

MOVE_ENUM:              market-avatar-flip | market-transformation-change |
                        um-mechanism-reveal | um-problem-framing | um-proprietary-naming |
                        angle-desire | angle-pain | angle-external-blame | angle-care-signaling |
                        angle-identity-belonging | angle-curiosity-secret |
                        offer-bundle | offer-guarantee | offer-urgency-scarcity | offer-price-anchor
                        (from the four differentiator levers: Market / UM / Angle / Offer;
                        angle-lever tags map to Weische's angle categories)

ROUTING_FLAG_ENUM:      structure_only | messaging_full | both
                        structure_only  = off-transformation crowdfunding: keep belief-sequence / container /
                                          ship-scaffolding; DISCARD specific claims/angles (pollutes messaging pile)
                        messaging_full  = in-transformation brand (DTC or brand crowdfunding): keep everything,
                                          especially claims/angles/proof — this is proven persuasion
                        both            = in-transformation crowdfunding: richest source; keep all

BELIEF_ID_ANCHORS:      problem-exists | problem-matters | past-solutions-failed |
                        mechanism-is-the-reason | product-delivers-transformation |
                        trust-the-brand-or-founder | it-will-ship |
                        its-worth-the-price | act-now
                        (open-with-anchors: prefer an anchor; if none fits, propose a new belief_id
                        AND set belief_confidence to 'low')

BELIEF_KIND_ENUM:       crowdfunding-specific | general-dr
                        crowdfunding-specific = beliefs that only appear in crowdfunding funnels
                                                (e.g. it-will-ship, trust-the-brand-or-founder as founder-credibility)
                        general-dr            = beliefs that appear in any DR funnel
```

## DETERMINISTIC SCAFFOLD (scripts + hooks — NOT agent honor system)

**Scripts (run by orchestrator, agents never do this work):**
1. `tools/funnel-assemble.js` (Plan 03-01) — pulls ads, normalizes destination URLs, clusters ads→LP,
   renders each LP with Playwright, emits one `funnel_package` per cluster.
2. `tools/crowdfund-fetch.js` (Plan 03-01/02) — SPA/Cloudflare render + Currency-B stat parser.
3. `tools/funnel-clean.js` (Plan 03-02) — section-marked verbatim cleaner; emits `[SECTION]` markers on
   structural boundaries; preserves `[REVIEW_LANGUAGE_START/END]` blocks verbatim; stays dumb (does NOT
   segment beliefs).
4. `tools/funnel-score.js` (Plan 03-02) — two-lane validation scorer; stamps `validation_lane[]` and
   `validation_strength{}`; never normalizes currencies.
5. `tools/funnel-store.js` (Plan 03-04) — writes structured JSON to `runs/<space>/funnels/`; RAG-ready,
   not vectors.

**Hooks (PostToolUse on each agent's Write — reject, don't trust):**
- `tools/hooks/inject-dr.js` — DR-file BUNDLER (NOT an auto-injection hook — no such mechanism
  exists; the Section Analyzer runs as a subagent where settings hooks don't fire). Run it to
  (re)generate `prompts/_generated/section-analyzer-dr-context.md`, the concatenated six-file
  bundle with attribution headers. The orchestrator EMBEDS this bundle into the Section Analyzer's
  spawn prompt — the analyzer never Reads it; the orchestrator regenerates the bundle if the DR
  sources changed.
- `tools/hooks/validate-analyzer.js` — PostToolUse reject on every Section Analyzer Write:
  - VERBATIM-SUBSTRING GATE: any `verbatim_refs[].text` not a verbatim substring of the cleaned funnel
    copy → REJECT (kills hallucinated/paraphrased verbatim).
  - OVERFLOW-BELIEF RULE: `belief_id` outside the 9-anchor set with `belief_confidence` ≠ 'low' → REJECT.
  - CLOSED-VOCAB REJECT: `execution_type` / `proof_tier` / `claim_type` / `move` off-enum → REJECT.
  - POSITION RULE: duplicate `position` values within a funnel or page-local position → REJECT.
  - SINGLE-FUNNEL DISCIPLINE: any birdseye-only field (`consensus`, `divergence`, `unusual`, pool
    reasoning) present on a belief record → REJECT.
