# PMF Build-State Map

## Orientation

PMF is a research-to-launch system for direct-response e-commerce. It operates on a locked phase spine numbered 0–8: Phases 0–4 are the research engine (map a space, theorize, study competition, study the customer, design the test); Phases 5–8 are the execution engine (hook test, build-a-brand, eval, iterate or destroy). Three-layer document discipline structures everything: Layer 1 is flow/workflow (what happens, in what order, why); Layer 2 is build briefs and agent prompts (theory-heavy specs, heuristics, classification rules); Layer 3 is technical implementation (APIs, scripts, storage wiring). The Inkleaf foldable e-ink tablet is the first product moving through this system, currently mid-Phase 0 with partial Phase 2/3 surface area explored ad hoc. This map is a feeder document for `gsd-new-project`: it tells the GSD setup which artifacts exist, which are missing, what is buildable now versus blocked, and how to split milestones without colliding with PMF's own research-phase numbering.

## THE INKLEAF PHASE-6 FOLD

### What ad hoc machinery exists

The Inkleaf e-ink run (2026-04-26 to 05-29) generated a substantial body of work that was never specced upfront. It exists as instance artifacts inside `runs/eink-tablets/`, produced through 14 mined session transcripts and 6 run-output clusters. The machinery breaks into three categories:

**Reusable process (extract and templatize):**
- `scripts/analyzer-framework.md` — the shared analyzer spine; corrections propagate from one file. The pattern of "one framework file, N agent instances read it" is the key reusability mechanism.
- `scripts/granular-analyzer-brief.md` — 13-section per-brand deep analysis with hard rules + self-audit checklist. The checklist/forbidden-reading structure is the reusable template pattern.
- `scripts/market-playbook-brief.md` — per-market synthesis with SYNTHESIS-block flagging discipline.
- `scripts/crowdfund-fetch.js` — Playwright SPA/Cloudflare bypass fetcher. Reusable for any SPA scrape.
- `scripts/adlib-one.js` — Meta Ad Library fetcher with `days_running` enrichment. Reusable.
- Agent brief template (role + inputs + rules + output schema + self-audit checklist + exact output path) — extracted as a pattern from the briefs; not yet a standalone doc.
- The 3-agent scan pattern: finder → roster checkpoint → parallel analyzers → aggregator → profile checkpoint.
- 2-pass corpus-then-synthesize discipline (rip verbatim first, synthesize second).
- Deliverable schemas (market-profile, market-opportunity, granular analysis, playbook, crowdfunding teardown A/B/C/D, deposit-funnel skeleton) — described in `run-retrospective.md` §3; not yet consolidated.

**Instance data (Inkleaf-specific, not directly reusable but valuable):**
- 31 brand records under `runs/eink-tablets/brands/`
- 10-brand marketing corpus under `runs/eink-tablets/marketing-corpus/{brand}/`
- 3 market profiles (faith, students, dumb-device) with Gate 1 evidence dossiers
- `market-opportunity.md` (4 adjacent markets analysis)
- 4 per-market playbooks (M1–M4)
- Category evolution profile + transformations flat map
- Crowdfunding corpus: 4+ campaigns with full 9-file schema each
- `lp-teardown.md` (deposit-funnel 3-stage skeleton)
- `deposit-pages-verbatim.md`
- `winning-message-analysis.md` (Daylight DC-1)

**Cruft (raw intermediate artifacts with no downstream use):**
- `runs/eink-tablets/adlibrary/` — 107 raw `.txt` + `.png` screenshot dumps from the Meta Ad Library sweep. These are raw scraper output. The resolved, enriched data lives in the per-brand records and granular analyses. The raw dump files have served their purpose. The duplication within this folder (e.g., `Boox.txt` + `Boox_adv.txt` + `boox_adv.txt` — multiple passes of the same brand under different casings and keyword variants) adds noise with no value. Candidate for deletion after confirming no undigested signal remains.
- `runs/eink-tablets/adlibrary/freewrite_adv_adv.txt` — double-suffixed artifact indicating a retry/error run. Cruft.
- `runs/eink-tablets/adlibrary/test_adv.txt` — test artifact. Cruft.
- `runs/eink-tablets/marketing-corpus/markets/M1-paper-replacement/_drafts/supernote-deep.md` — abandoned draft. Cruft if superceded by the final playbook.
- `runs/eink-tablets/brands.md` (root-level flat list) — superseded by the per-market competitive sets and individual `brands/*.md` records. Check before deleting.

- `runs/eink-tablets/scripts/sw-login.js`, `sw-sweep.js`, `explore-typeahead.js` — SimilarWeb scripts. Check if any sweep data was actually used; if not, cruft.

## NUMBERING / NAMING RECONCILIATION

### The collision

Four numbering schemes are in use simultaneously:

1. **PMF research Phases 0–8.** Locked. These are the research workflow stages defined in `workflow.md`. They describe what to research and when. Do not renumber.

2. **`flow.md` lettered phases (implied A–H or numbered differently).** `flow.md` uses headings like "Phase 0," "Phase 1," "Phase 2" but its Phase 1 covers what `workflow.md` calls the finder/sketching work (map a space), and its Phase 2 covers "Sketch Your Competition" — which is closer to `workflow.md`'s Phase 0–1 work. The numbering in `flow.md` does not match `workflow.md`. This is an internal inconsistency, not a new scheme.

3. **Eink-tablets run "Phase 0–3" shorthand.** The Inkleaf handoffs (`handoff.md`) refer to "Phase 0 steps 2–3" meaning steps within PMF's research Phase 0. This is a sub-step reference, not a parallel numbering system.

4. **GSD build units ("stages").** The `handoff-phase3-voc-build.md` §0 establishes the rule explicitly: GSD's build execution units are called **"stages"** to avoid collision with PMF's Phase 0–8 research vocabulary. "Stage" = a GSD deliverable unit. "Phase" = PMF research workflow step. These must never be conflated.

### Locked resolution rule

- **"Phase" = PMF research workflow step, numbers 0–8, defined in `workflow.md`.** Immutable. Any doc using "Phase N" refers to this.
- **"Stage" = GSD build unit.** Use this in all GSD planning, ROADMAP.md, milestone docs. Never use Phase numbers for GSD stages.
- `flow.md` numbering collision: treat `flow.md` as a sketch layer only. Canonical workflow authority is `workflow.md`. When `flow.md` and `workflow.md` diverge, `workflow.md` wins.
- "Milestone" = a GSD grouping of stages for delivery. M1 = research engine stages; M2 = execution engine stages. These are delivery buckets, not Phase numbers.
