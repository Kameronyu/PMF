# Phase 17 — UI Design Contract (Arduview instance · "Glasshouse")

> Visual and interaction contract for the Arduview LP-builder instance. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note (D-02):** This is the per-instance *visual contract* the reusable LP-builder consumes. It is the Arduview instance — bootstrapped from the already-locked "Glasshouse" system, not invented here. The builder is product-agnostic; this file is one of its inputs.
>
> **Authority (D-10):** Once this file exists it is the **single design source**. It SUPERSEDES `STYLE-LOCK.md` and `BUILD-FEEDBACK.md §1`; those become inputs-only and are never re-read by the builder. This document must stand alone — everything the builder and checker need is here.
>
> **Reuse rule (D-03):** The builder composes/edits HTML against the prebuilt, locked `styles.css`. It needs class *names* (below), never the CSS body. It NEVER authors or re-authors CSS.
>
> **Decision surface vs. implementation reference (read first):** This contract separates two things. The **declared contract** (Spacing Scale, Typography tables below) is the builder's *decision surface* — the small set of role-families/tokens the builder picks from when it assigns a class to an element. The **CSS implementation reference** subsections record the locked per-class pixel/weight values that already live in `styles.css`. The builder *reads* the reference to know what a class will render, but it never *chooses*, re-authors, or expands those values. The implementation reference is NOT part of the declared contract and is NOT what the dimension caps (≤4 sizes, ≤2 weights, multiples-of-4) apply to.

## Spacing Scale

### Declared contract (builder decision surface)

The locked system is **fluid** (`clamp()`-based), not a fixed 8-point token set. These are the four locked spacing primitives the builder chooses from — it applies these tokens/classes, never invents one-off margins.

| Token | Value | Usage |
|-------|-------|-------|
| `--gutter` | `clamp(20px, 5vw, 64px)` | Horizontal page padding (`.wrap`, `.nav`, `.footer`) |
| `--section-y` | `clamp(72px, 12vh, 160px)` | Vertical rhythm between full sections (`.section`); `.section--tight` overrides to `clamp(40px, 7vh, 88px)` |
| `--maxw` | `1120px` | Content max width; hero overrides to `1320px` |
| split/founder/offer gap | `clamp(28px, 5vw, 72px)` | Two-column column gap |

Exceptions: spacing is **fluid by design** (responsive viewport units) — the "multiples of 4" rule does not apply to the `clamp()` tokens above. The builder must reuse these tokens/classes and not introduce arbitrary pixel margins.

> **CSS implementation reference — builder does not choose these (NOT part of the declared spacing contract).** The locked `styles.css` resolves the tokens above into discrete margins/paddings on specific components (e.g. `8 / 14 / 18 / 22 / 24 / 26 / 28 / 34 / 48 / 56 px` rhythm steps already baked into `.eyebrow`, `.pills`, `.chips`, `.feature`, `.letter`, `.price-row`, `.reassure`, etc.). These are pre-authored implementation details of the locked stylesheet, recorded here only so the spec stands alone (D-10). They are NOT a token set the builder selects from, NOT extensible, and the multiples-of-4 rule does not apply to them — the builder never authors a margin; it applies a class and the locked CSS supplies the value.

## Color

60/30/10 reading: black/white surfaces alternate as the dominant frame; greyscale ramp is the secondary; cyan glow is the strict <5% accent. **The product photos carry the real color (TE principle) — cyan is punctuation only.**

| Role | Value | Usage |
|------|-------|-------|
| Dominant surface — dark | `#0a0a0b` (`--black`) | Hero, "wow"/hardware bands, deposit page, footer. Where transparency reads. |
| Dominant surface — light | `#ffffff` (`--white`) + `#f6f6f4` (`--paper`) | Editorial/spec bands ("what it is", FAQ) |
| Raised dark surface | `#131316` (`--black-2`) | Founder band, raised panels |
| Secondary text on light | `#44444a` (`--grey-1`) | Muted copy on white |
| Secondary text on dark | `#b8b8be` (`--grey-2`) | Muted copy on black |
| Hairlines | `#e6e6e3` (`--line-light`) / `#26262b` (`--line-dark`) | Rules, borders |
| **Accent — screen-glow cyan** | `#2be5ff` (`--glow`) | RESERVED list below |
| Accent dim | `rgba(43,229,255,0.14)` (`--glow-dim`) | Focus rings, glow shadows |

**Accent (cyan `#2be5ff`) reserved for — explicit list, <5% of any view, one highlight per section:**
- the dot motif (pill status dot, eyebrow lead-rule, FAQ `// ` markers, `lockins`/`chip` ticks)
- the in-copy highlight class `.hl` (exactly one per section)
- interactive accent: button fill, input focus ring/border, link hover glow
- eyebrows, proof-chip values, `save-flag` background, wordmark `®`

**Accent NEVER on:** body copy, headline body text, decorative fills beyond the above. Never a second accent color (no second hue, ever).

Destructive: **none.** This phase has no destructive actions (LP + deposit funnel only). No destructive palette declared.

## HARD CONSTRAINTS — page-craft rules (ported VERBATIM from BUILD-FEEDBACK §1, D-11)

These are non-negotiable, machine-checkable rules. A ui-checker MUST be able to enforce each. They override any softer guidance above on conflict.

### HARD-1 — Context-aware accent: NO black-on-black, EVER (recurred twice — highest-priority rule)
The accent/highlight color MUST be **context-aware** — readable on BOTH light and dark surfaces. **No text on a same-value background, ever.**
- Symptom 1 that triggered this rule: hero headline highlight "see right through" rendered black-on-black.
- Symptom 2 (same bug): deposit "$5 deposit" highlight black-on-black.
- Root cause: the highlight class defaulted to dark ink and was patched for only *some* dark containers.
- Enforcement: any `.hl` (or equivalent highlight) inside a dark surface (`.hero`, `.section--dark`, `.deposit-hero`, `--black`/`--black-2` bg) MUST resolve to cyan glow (`color: var(--glow)`, no dark border) — never inherit `--ink`. On light surfaces `.hl` is ink with a cyan underline. The locked CSS already encodes both branches; the builder must not place a highlight in a container where the cascade leaves it same-value.
- **Checker test:** for every highlighted text run, computed text color ≠ computed background color (contrast must clear, not collapse).

### HARD-2 — CTA legibility: black-on-accent, readable face (NOT the display font)
CTA text MUST be **black (`#000`) on the cyan accent fill**, in a **readable face — IBM Plex Mono (`var(--mono)`), weight 500**. NEVER the Doto dot-matrix/display font; it is unreadable at button size.
- Locked: `.btn { background: var(--glow); color: #000; font-family: var(--mono); font-weight: 500; }`.
- **Checker test:** every primary CTA button has accent background, black text, and a non-Doto font-family.

### HARD-3 — 17px body minimum
Base body font size is **17px minimum**. Secondary-text contrast is raised: greys darkened on light (`--grey-1 #44444a`), lightened on dark (`--grey-2 #b8b8be`). Leads bumped up. No body or secondary copy below 17px equivalent except the conventional fine-print classes (`.cta__fine` 13px, `.nav__meta` 12px, `.paywall`/`.offer__secure` 12px, `.chip` 13px) which are deliberate sub-body labels.
- **Checker test:** `body { font-size: 17px }`; no primary reading text under 17px.

### HARD-4 — Per-page fold budget (state explicitly; never leave to the builder)
**LP (`index.html`) hero above the fold:** title + CTA + video. Subheader sits BELOW the fold. **Video is the centerpiece — ≥ half the hero, bigger than instinct.**
**Deposit (`deposit.html`) above the fold:** image + title + CTA + the value (51% off). Image beside the title (no scroll to see it). CTA up top. Description may drop below.
- **Checker test:** above-the-fold DOM of each page contains exactly the budgeted elements (and the hero video occupies ≥50% of hero area on LP).

### HARD-5 — Explicit element order per breakpoint (state both, never infer)
- **LP desktop:** headline + CTA left, video right; subheader full-width below.
- **LP mobile:** title → CTA → video → subheader. Hero media full-bleed (edge-to-edge) on mobile.
- **Deposit desktop:** copy (tag → title → price-row → value → CTA → secure note) left, image right.
- **Deposit mobile:** image on top (capped height) → title → price → CTA → (reassurance below).
- Breakpoints: `≤860px` (stack two-cols, deposit image to top, hero stacks + full-bleed video), `≤520px` (CTA form goes column, full-width input + button).
- **Checker test:** DOM/CSS order matches the stated order at each breakpoint.

### HARD-6 — Copy hierarchy / risk-reversal (deposit), no repetition
Tighten copy but **do not delete any risk-reversal move** (deposit credits toward price; fully refundable; 30-day money-back). **Do not repeat yourself** — each risk-reversal point appears once, in one zone. **Value (51% off) above the fold; reassurance as the user scrolls.**
- **Checker test:** each risk-reversal point appears exactly once; the 51% value is above the fold.

### HARD-7 — Additive restraint (BUILD-FEEDBACK §2, builder behavior, D-05)
The builder renders the brief in the locked visual system and adds **no customer-facing content the brief didn't ask for — in words or in chrome.** Every visible word comes from the copy brief (PAGE-SPEC). No image captions/figcaptions, no invented eyebrows/kickers, no badge labels, no helper microcopy, no "what is this" tags, no decorative UI the brief didn't call for.
- The rule applies ONLY to customer-facing/visible content. Non-visible strings are unconstrained: `alt`, `aria-*`, `data-*`, class names, file names, code comments — be descriptive there.
- Conventional unavoidable functional strings (form placeholders e.g. `your email`, button states) are fine, kept minimal.
- Also avoid the generic visual AI-tells: gratuitous gradients, emoji-as-icons, three-identical-feature-cards, centered-everything defaults. (The locked Glasshouse system already fights these — `figcaptions` are CSS-hidden via `.frame__tag { display: none }`.)
- **Checker test:** no figcaptions rendered; no visible string absent from the PAGE-SPEC copy brief; no decorative chrome beyond the locked component set.

> **REJECTED — do NOT encode:** BUILD-FEEDBACK §3 image-numbering convention is rejected as brittle. Do not add image-N labels, numbering, or position semantics to the contract.
